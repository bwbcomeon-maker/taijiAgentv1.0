# 可访问性与长时间工作体验检查清单

## 可访问性检查

- accessible name：每个交互控件都有可读名称。
- 图标按钮：有 aria-label、title 或可见文字，不能只有图标。
- 表单 label：每个字段都有 label，不能只依赖 placeholder。
- 错误关联：错误信息和对应字段有关联，读屏和视觉上都能理解。
- 焦点状态：键盘焦点清楚可见。
- Tab 顺序：符合视觉顺序和任务顺序。
- 弹窗焦点管理：打开后焦点进入弹窗，关闭后回到触发位置或合理位置。
- Escape / Cancel 行为：弹窗、菜单、抽屉可用 Escape 或清晰按钮关闭。
- 颜色非唯一信号：状态不能只靠颜色表达。
- 对比度：文本、边框、焦点、禁用状态有合理对比。
- 点击目标大小：常用控件和破坏性控件的点击区域不能过小。
- 禁用状态解释：用户能理解为什么不可操作。
- 键盘完成主路径：键盘用户可以完成核心任务。
- 减少动态效果：尊重 reduced motion，避免不可关闭的频繁动画。

## 长时间工作体验检查

- 字号：正文、表格、表单适合长时间阅读。
- 行高：不会过挤或过松。
- 行宽：长文本不过长，避免横向扫读疲劳。
- 信息密度：密度服务于效率，不造成主次混乱。
- 视觉噪音：避免过多强边框、强阴影、大面积高亮和同权重按钮。
- 滚动体验：滚动区域明确，避免多层嵌套滚动陷阱。
- 上下文稳定：列表选中项、详情页、编辑状态、滚动位置尽量稳定。
- 长列表扫描：提供足够标题、分组、状态、间距或行内线索。
- 长时间阅读疲劳：避免刺眼对比、过亮背景、过密文本和频繁动画。
- 常用操作可见性：高频操作不应藏在深层菜单中。

## 手动检查兜底方式

当自动化工具不可用时，至少手动检查：

- 用键盘 Tab 完成核心路径。
- 检查所有图标按钮是否有可见文本或 aria-label。
- 检查表单 label、错误提示和提交反馈。
- 检查弹窗/菜单的焦点进入、Escape 关闭和焦点返回。
- 检查默认、空、加载、错误、禁用和破坏性确认状态。
- 检查 30 秒连续阅读和操作时是否出现明显视觉疲劳或上下文丢失。

## 自动化检查缺失时如何标记“未验证”

- 如果没有 axe、@axe-core/playwright、Lighthouse 或类似工具，写“自动化可访问性：未验证。原因：项目未配置相关工具。”
- 如果工具存在但无法运行，写“自动化可访问性：未验证。原因：……”并贴出关键错误。
- 手动检查可以写“人工语义检查已执行”，但不能写“自动化可访问性通过”。
- 未执行截图、视觉回归或浏览器键盘检查时，必须进入最终报告的“未验证项目”。

# 前端 UX 质量评分规则

## 前端完成定义

前端完成不是“代码能编译”。前端完成必须同时满足：

- 用户目标清楚，页面主内容突出。
- 代码中存在的用户可感知能力都有可见、可发现、可访问的 UI 入口。
- 主路径经过真实浏览器验证，或明确标记为“未验证”。
- 空、加载、错误、成功、禁用、破坏性操作状态已经检查。
- 基本可访问性、视觉层级、响应式布局和长时间工作体验已经检查。
- 自动化检查已运行，或明确写为“未验证”。
- 输出中文《前端 UX QA 报告》。

## 功能完整性规则

- 从代码层和页面层双向检查功能是否完整。
- 代码层检查 API endpoint、mutation、reducer、state action、event handler、command、service function、route、component prop、feature flag。
- 页面层检查按钮、菜单项、图标按钮、上下文菜单、快捷键说明、空状态入口、错误恢复入口、确认/撤销/取消路径、成功反馈。
- 任何用户可感知能力都必须能被普通用户通过 UI 发现和操作。

## 功能实现与 UI 入口一致性规则

- 如果代码存在某个 action、handler、mutation、command 或业务能力，但页面没有可见、可发现、可访问入口，标记为 P1。
- 如果主流程必须依赖隐藏快捷键、未说明手势、开发者工具或 URL hack，标记为 P0 或 P1。
- 如果入口存在但没有 accessible name、位置不符合预期、窄屏丢失、禁用无解释或操作无反馈，至少标记为 P2；影响核心路径时标记为 P1。

## CRUD / UI 入口规则

- 创建、查看、编辑、删除、搜索、筛选、排序、分页、批量操作必须端到端检查。
- CRUD 能力必须有清晰入口、执行反馈、错误恢复和必要状态。
- 删除、覆盖、清空、批量修改等破坏性操作必须有确认、取消、撤销或恢复路径。

## 信息架构规则

- 主内容是用户来到页面最想看、最需要判断或最需要操作的内容。
- 辅助内容用于解释上下文，不能压过主内容。
- 高级内容包括元数据、低频字段、调试信息、历史信息、详细配置和二级操作。
- 不要把所有字段平铺成同等视觉权重。
- 不要因为数据存在就全部展示。
- 对高级信息优先使用折叠、详情区、分组、标签页、抽屉或渐进披露。
- 列表、详情、编辑、删除、筛选、分页、滚动过程中必须保持用户上下文稳定。

## 主内容、辅助内容、高级内容区分规则

| 类型 | 判断问题 | 合格表现 |
|---|---|---|
| 主内容 | 用户最需要看什么、做什么、判断什么 | 页面首屏明确、视觉权重最高、主操作可见 |
| 辅助内容 | 什么信息帮助用户理解主内容 | 位于次级层级，不抢焦点 |
| 高级内容 | 什么是低频、元数据、调试或二级配置 | 默认折叠或分组展示 |

## 状态覆盖规则

必须检查：

- 空状态：是否有解释、下一步入口和恢复方式。
- 加载状态：是否避免布局跳动，是否反馈进行中。
- 错误状态：是否具体、可操作，是否有重试或恢复路径。
- 成功状态：是否让用户知道操作完成。
- 禁用状态：是否有原因解释。
- 破坏性状态：是否有确认、取消、撤销或恢复。

## 破坏性操作安全规则

- 删除、覆盖、清空、归档、批量修改等操作不能太容易误触。
- 破坏性操作要清楚但不过度突出。
- 必须有确认或恢复机制；缺失时通常为 P1。
- 高风险操作不能与常规主操作混在一起造成误触。

## 真实浏览器测试规则

- 不得只读代码就声称前端完成。
- 优先使用 Playwright MCP，其次使用项目已有 Playwright E2E，再其次使用 dev server / preview / Storybook。
- 必须真实打开页面，执行点击、输入、滚动、键盘、焦点和主要流程检查。
- 如果无法使用真实浏览器，报告必须写“真实浏览器测试：未验证。原因：……”，不能写“通过”。

## 可访问性规则

- 每个交互控件必须有 accessible name。
- 图标按钮必须有 aria-label 或可见文字。
- 表单字段必须有 label。
- 错误信息必须和字段有关联。
- Tab 顺序必须符合视觉和任务顺序。
- 焦点状态必须清晰可见。
- 弹窗打开后焦点进入弹窗，关闭后焦点回到合理位置。
- 弹窗、菜单、抽屉可用 Escape 或清晰按钮关闭。
- 颜色不能是唯一信息来源。
- 键盘用户必须能完成核心路径。

## 长时间工作体验规则

- 字号、行高、行宽适合长时间阅读。
- 信息密度不过高，对比不过低也不过刺眼。
- 滚动区域清楚，列表和详情区保持上下文。
- 当前选中项、焦点和状态变化清晰但不过度刺激。
- 避免大面积高亮色、强边框、强阴影、频繁动画和频繁弹窗。
- 常用操作不能藏在深层菜单中。
- 页面整体必须像可长期工作的产品界面，而不是临时调试界面。

## 视觉设计质量规则

- 主内容最突出，标题和当前上下文清楚。
- 主操作容易发现，次要操作不干扰主操作。
- 间距一致，对齐有意图。
- 字号、字重、颜色形成层级。
- 列表、表格、卡片便于扫描。
- 空、加载、错误状态看起来像产品状态，而不是开发占位符。
- 页面在常见桌面和窄屏视口下可用。

## 响应式规则

- 关键入口不能在窄屏或中等视口丢失。
- 文本不能溢出按钮、卡片、表格或导航。
- 固定工具栏、侧边栏、详情面板不能遮挡主内容。
- 滚动区域必须可理解，不能形成嵌套滚动陷阱。

## 自动化检查规则

- 优先运行项目已有 lint、typecheck、test、test:unit、test:e2e、test:playwright、test:ui、build。
- 如果项目已有 axe、@axe-core/playwright、Lighthouse、Storybook、Chromatic 或视觉回归工具，优先使用现有工具。
- 不要发明不存在的命令。
- 不要自动安装依赖。
- 不要自动修改 package.json、lockfile 或 CI。
- 命令不存在写“未配置”或“未验证”，不能写“通过”。

## “未验证”处理规则

- 检查没有执行时必须写“未验证”。
- 无法启动项目、无法打开浏览器、无法截图、无法运行 a11y 自动化或视觉回归时，必须写清原因。
- 未验证项不得隐藏在备注里，必须进入最终报告的“未验证项目”。

## 严重程度定义

P0 - 阻塞问题：
- 主用户路径无法完成。
- 页面崩溃。
- 存在明显数据丢失风险。
- 常见操作后用户无法恢复。
- 核心内容不可见、不可访问或不可操作。
- 用户无法通过可见 UI 完成核心任务。

P1 - 严重问题：
- 代码中实现了用户能力，但页面没有可发现 UI 入口。
- CRUD 能力缺少可见入口。
- 删除、覆盖、清空、批量修改等破坏性操作没有确认、撤销或恢复。
- 主操作隐藏、歧义、不可靠。
- 核心控件存在严重可访问性问题。
- 页面布局明显阻碍正常使用。
- 重要空状态、错误状态、加载状态缺失。
- 主流程必须依赖隐藏快捷键、未说明手势或开发者才知道的路径。

P2 - 质量问题：
- 视觉层级差。
- 信息密度过高。
- 辅助信息抢占主内容。
- 间距、标签、交互不一致。
- 焦点状态弱。
- 键盘体验勉强可用但效率差。
- 长时间工作观看不舒适。
- 响应式布局存在明显问题但不完全阻塞。

P3 - 打磨问题：
- 轻微对齐问题。
- 文案可优化。
- 微交互可优化。
- 局部视觉细节可改进。
- 轻微动画或过渡问题。

## 坏结果与可接受结果示例

坏结果：

- 对话列表实现了删除逻辑，但没有删除按钮。
- 页面展示所有字段，主内容不突出。
- 删除操作没有确认。
- 错误状态只显示“出错了”，没有恢复路径。
- 图标按钮没有 aria-label。
- 列表密度过高，长时间阅读疲劳。
- 浏览器测试没有运行，但报告写成通过。

可接受结果：

- 对话列表有明确删除入口、确认弹窗、取消路径、成功反馈和错误恢复。
- 页面优先展示当前用户最需要处理的内容，高级信息折叠。
- 主路径经过 Playwright 或真实浏览器验证。
- 未执行的检查明确写为“未验证”。

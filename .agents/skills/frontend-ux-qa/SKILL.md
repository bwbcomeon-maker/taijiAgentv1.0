# Frontend UX QA Skill

## 何时必须使用本 Skill

只要任务涉及以下任意内容，就必须使用本 Skill：

- 前端代码
- 页面布局
- UI 组件
- CSS、Tailwind、样式、主题、设计 token、间距、字体、颜色、响应式布局
- 导航
- 表单
- 列表、表格、卡片、仪表盘、侧边栏
- 弹窗、抽屉、菜单、下拉框、提示框
- CRUD 流程
- 用户交互
- 空状态、加载状态、错误状态、成功状态、禁用状态
- 删除、归档、覆盖、清空、批量操作等破坏性操作
- 可访问性
- 视觉优化
- 前端测试
- Playwright、Storybook、截图、视觉回归
- 任何用户能看到或操作的内容

不得因为任务看起来很小就跳过本 Skill。

## 核心原则

前端任务不是在代码编译成功时完成。

前端任务只有在以下条件满足后，才可以被认为完成：

- 用户目标清楚。
- 页面主内容清楚、突出、容易理解。
- 辅助内容和高级内容没有干扰主任务。
- 所有已经在代码中实现的能力，都有可见、可发现、可访问的 UI 入口。
- 主要用户路径已经在真实浏览器中验证，或明确标记为“未验证”。
- 空状态、加载状态、错误状态、成功状态、禁用状态、破坏性操作状态都被检查。
- 键盘操作、焦点状态、图标按钮名称、表单标签、弹窗焦点管理等基本可访问性被检查。
- 页面适合长时间工作观看，不会因为密度、对比、滚动、布局不稳定、信息平铺导致疲劳。
- 相关自动化检查已经运行，或明确标记为“未验证”。
- 输出了中文《前端 UX QA 报告》。

硬规则：

如果代码中存在某个用户可感知的功能、API、状态更新、action、reducer、handler、mutation、command 或业务能力，但页面中没有可见、可发现、可访问的 UI 控件入口，则该功能未完成。

该问题必须至少标记为 P1。

示例：

- 如果存在 deleteConversation，但对话列表没有删除按钮、删除菜单或明确删除入口，标记为 P1。
- 如果存在 renameConversation，但只能通过未说明的快捷键触发，标记为 P1。
- 如果存在 archive、duplicate、export、share 等能力，但用户无法通过可见控件发现，标记为 P1。
- 如果主流程必须依赖隐藏手势、未文档化快捷键或开发者才知道的入口，标记为 P0 或 P1。
- 如果主任务完全无法完成，标记为 P0。
- 如果检查没有执行，必须写“未验证”，不能写“通过”。

## 必须执行的工作流

每次前端任务都必须执行以下步骤。

### 1. 变更分类

在编码前或检查前，先识别：

- 变更了哪些页面。
- 变更了哪些组件。
- 影响了哪些用户路径。
- 影响了哪些数据、状态、API、action、mutation、reducer 或 handler。
- 用户的主要目标是什么。
- 页面主内容是什么。
- 页面辅助内容是什么。
- 哪些内容属于高级信息、低频信息或元数据。
- 是否存在创建、查看、编辑、删除、搜索、筛选、排序、批量操作。
- 是否存在破坏性操作。
- 是否存在长时间工作使用场景。
- 是否存在可访问性风险。
- 是否存在响应式布局风险。

### 2. 建立功能契约

使用 `references/feature-contract-template.md`。

必须创建或更新一个功能契约表，至少包含：

- 能力
- 数据/API/状态是否存在
- UI 入口是否存在
- 用户反馈是否存在
- 错误处理是否存在
- 空/加载/禁用状态是否存在
- 键盘/可访问性支持是否存在
- E2E/浏览器测试是否存在
- 状态：通过 / 失败 / 未验证
- 备注

硬规则：

- 每个用户可感知能力必须有可见、可发现、可访问的 UI 入口。
- CRUD 能力必须端到端检查。
- 如果代码支持某个操作，但 UI 不暴露该操作，状态必须写“失败”。
- 如果操作只能通过隐藏快捷键、未说明手势、开发者工具或 URL hack 触发，状态必须写“失败”。
- 如果没有执行检查，状态必须写“未验证”。

### 3. 检查功能完整性

检查代码和 UI 是否一致。

必须从代码层和页面层双向检查：

从代码层检查：
- API endpoint
- mutation
- reducer
- state action
- event handler
- command
- service function
- utility function
- route
- component prop
- feature flag

从页面层检查：
- 是否有按钮
- 是否有菜单项
- 是否有图标按钮
- 是否有上下文菜单
- 是否有快捷键说明
- 是否有空状态入口
- 是否有错误恢复入口
- 是否有确认/撤销/取消路径
- 是否有成功反馈

重点发现：

- 实现了功能但没有入口。
- 入口存在但不可发现。
- 入口存在但没有 accessible name。
- 入口存在但位置不符合用户预期。
- 入口存在但移动端/窄屏丢失。
- 入口存在但禁用后没有解释。
- 入口存在但没有反馈。
- 删除等破坏性操作没有确认、撤销或恢复。

### 4. 检查信息架构

对每个变更页面明确区分：

- 主内容：用户来到这个页面最想看、最需要判断或最需要操作的内容。
- 辅助内容：帮助理解主内容的上下文。
- 高级内容：元数据、低频字段、调试信息、历史信息、详细配置、二级操作。

规则：

- 不要把所有字段平铺成同等视觉权重。
- 不要因为数据存在就全部展示。
- 不要让辅助信息压过主内容。
- 不要让用户必须在大量同权重内容中扫描。
- 不要让页面像数据库字段直接 dump 出来。
- 优先突出主内容、主操作和下一步动作。
- 对高级信息使用折叠、详情区、分组、标签页、抽屉或渐进披露。
- 在列表、详情、编辑、删除、筛选、分页、滚动过程中保持用户上下文稳定。

### 5. 检查 UX 启发式问题

必须检查：

- 系统状态是否可见。
- 加载、保存、删除、失败、成功是否有清晰反馈。
- 标签、按钮、菜单是否符合用户心智模型。
- 控件、文案、间距、交互是否一致。
- 用户是否能取消、撤销、返回、恢复。
- 破坏性操作是否有确认或恢复。
- 常用操作是否可见。
- 用户是否被迫记忆隐藏入口。
- 错误是否可预防。
- 错误信息是否具体、可操作。
- 页面是否美观、克制、重点明确。
- 键盘操作是否合理。
- 响应式布局是否可用。
- 是否适合长时间工作。

### 6. 检查长时间工作体验

必须判断页面是否适合连续使用。

检查：

- 字号是否适合长时间阅读。
- 行高是否舒适。
- 行宽是否过长。
- 对比度是否过低或过刺眼。
- 信息密度是否过高。
- 间距是否稳定。
- 滚动区域是否清楚。
- 长列表、编辑区、详情区是否保持上下文。
- 侧边栏、顶部栏、详情面板是否稳定。
- hover、focus、active 状态是否清晰但不过度刺激。
- 是否存在大面积高亮色、强边框、强阴影、频繁动画。
- 是否存在过多同权重按钮。
- 是否存在频繁弹窗打断。
- 是否需要用户反复切换页面才能完成连续工作。
- 是否适合一小时以上持续查看和操作。

以下情况必须标记为问题：

- 控件过于拥挤。
- 点击目标过小。
- 文本过密。
- 主次内容混在一起。
- 视觉层级不清。
- 列表没有足够扫描线索。
- 当前选中项不明显。
- 操作后上下文丢失。
- 频繁跳页或滚动位置重置。
- 长流程被不必要弹窗打断。
- 常用操作被藏在深层菜单中。
- 页面整体像临时调试界面，而不是可长期工作的产品界面。

### 7. 检查可访问性

必须检查：

- 每个交互控件都有 accessible name。
- 图标按钮有 aria-label 或可见文字。
- 表单字段有 label。
- 错误信息和字段有关联。
- Tab 顺序符合视觉和任务顺序。
- 焦点状态清晰可见。
- 弹窗打开后焦点进入弹窗。
- 弹窗关闭后焦点回到合理位置。
- 弹窗、菜单、抽屉可用 Escape 或清晰按钮关闭。
- 颜色不是唯一信息来源。
- 禁用状态有解释。
- 删除等破坏性操作有明确提示。
- 点击目标不过小。
- 文本对比度和控件对比度合理。
- 减少动态效果偏好被尊重。
- 键盘用户可以完成核心路径。

如果项目存在 axe、@axe-core/playwright、Lighthouse、Lighthouse CI 或类似工具，优先使用现有工具。

如果这些工具不存在，不要安装依赖，必须将自动化可访问性检查标记为“未验证”，同时执行人工语义检查并给出接入建议。

### 8. 必须做真实浏览器测试

不得只通过阅读代码来判断前端任务完成。

必须优先使用可用的真实浏览器路径：

1. Playwright MCP
2. 项目已有 Playwright E2E 测试
3. 项目 dev server / preview server + 浏览器交互
4. Storybook / 组件预览
5. 截图审查
6. 代码审查兜底

如果可以使用 Playwright MCP，必须让 Codex 实际打开页面，通过页面结构快照或浏览器工具完成点击、输入、滚动、键盘操作、截图和流程验证。

每个受影响用户路径至少检查：

- 打开页面。
- 查看默认状态。
- 执行主操作。
- 如适用，执行创建。
- 如适用，执行查看。
- 如适用，执行编辑/重命名。
- 如适用，执行删除/归档/清空等破坏性操作。
- 检查取消路径。
- 检查错误恢复路径。
- 检查空状态。
- 检查加载状态。
- 检查禁用状态。
- 检查成功反馈。
- 检查键盘操作。
- 检查焦点状态。
- 检查响应式布局。
- 截图关键状态，或说明无法截图。

硬规则：

如果不能执行真实浏览器测试，必须在最终报告中写：

“真实浏览器测试：未验证。原因：……”

不得写成“已验证”。

### 9. 视觉 QA

检查：

- 主内容是否最突出。
- 页面是否有清晰标题和当前上下文。
- 主操作是否容易发现。
- 次要操作是否不会干扰主操作。
- 危险操作是否清楚但不过度突出。
- 间距是否一致。
- 对齐是否有意图。
- 字号、字重、颜色是否形成层级。
- 列表、表格、卡片是否便于扫描。
- 空状态是否有帮助。
- 加载状态是否避免布局跳动。
- 错误状态是否具体可操作。
- 页面是否像成品产品，而不是调试页面。
- 页面是否在常见视口宽度下可用。
- 是否有截图对比或视觉回归测试。

如果项目存在截图测试、Playwright toHaveScreenshot、Storybook、Chromatic、视觉回归工具，则使用现有工具或建议补充测试。

如果不存在，不要自动安装，报告为“视觉回归：未验证”，并给出建议。

### 10. 自动化检查，但不要自动乱装依赖

先检查项目已有脚本和文档，例如：

- package.json scripts
- README
- AGENTS.md
- CONTRIBUTING
- docs
- CI 配置

优先运行项目已有命令，例如：

- lint
- typecheck
- test
- test:unit
- test:e2e
- test:playwright
- test:ui
- build
- storybook test
- visual test

规则：

- 不要发明不存在的命令。
- 不要自动安装依赖。
- 不要自动修改 package.json。
- 不要自动修改 lockfile。
- 如果检查失败，判断是否与本次改动相关。
- 如果检查无法执行，写“未验证”。
- 如果命令不存在，写“未配置”或“未验证”，不能写“通过”。

如果项目已有 Playwright：
- 为关键路径补充或更新 E2E 测试。
- 至少覆盖功能入口是否存在。
- 至少覆盖主要 CRUD 路径。
- 至少覆盖破坏性操作的确认/取消/反馈。
- 尽量覆盖空状态、错误状态、禁用状态。

如果项目已有 @axe-core/playwright：
- 为受影响页面补充或更新可访问性检查。

如果项目已有 Lighthouse 或 Lighthouse CI：
- 运行或建议运行相关可访问性、性能、最佳实践检查。

### 11. 复杂前端工作使用 Subagents

如果是复杂 UI、核心路径、大页面重构、PR review、设计系统改动或影响多个页面的前端变更，必须在可用时显式使用 subagents。

建议使用六个子审查角色：

1. 功能完整性 QA
2. UX 与信息架构审查
3. 视觉设计与布局审查
4. 可访问性与键盘导航审查
5. 长时间工作体验审查
6. 测试覆盖与回归风险审查

每个 subagent 都应检查代码；如果环境允许，还应通过浏览器实际操作页面。

必须等待所有 subagent 返回后，再合并结论。

如果 subagents 不可用，报告为“Subagents：未验证/不可用”，并由当前 Codex 进行人工合并审查。

### 12. 严重程度定义

使用以下严重程度：

P0 - 阻塞问题：
- 主用户路径无法完成。
- 页面崩溃。
- 存在明显数据丢失风险。
- 常见操作后用户无法恢复。
- 核心内容不可见、不可访问或不可操作。
- 用户无法通过可见 UI 完成核心任务。

P1 - 严重问题：
- 代码中实现了用户能力，但页面没有可发现 UI 入口。
- CRUD 能力缺少可见入口。
- 删除、覆盖、清空、批量修改等破坏性操作没有确认、撤销或恢复。
- 主操作隐藏、歧义、不可靠。
- 核心控件存在严重可访问性问题。
- 页面布局明显阻碍正常使用。
- 重要空状态、错误状态、加载状态缺失。
- 主流程必须依赖隐藏快捷键、未说明手势或开发者才知道的路径。

P2 - 质量问题：
- 视觉层级差。
- 信息密度过高。
- 辅助信息抢占主内容。
- 间距、标签、交互不一致。
- 焦点状态弱。
- 键盘体验勉强可用但效率差。
- 长时间工作观看不舒适。
- 响应式布局存在明显问题但不完全阻塞。

P3 - 打磨问题：
- 轻微对齐问题。
- 文案可优化。
- 微交互可优化。
- 局部视觉细节可改进。
- 轻微动画或过渡问题。

### 13. 必须输出最终《前端 UX QA 报告》

在声称前端任务完成前，必须使用 `references/ux-audit-report-template.md` 输出中文报告。

报告必须包含：

- 状态：完成 / 带限制完成 / 未完成 / 阻塞
- 变更范围
- 主要用户目标
- 主内容 / 辅助内容 / 高级内容
- 已测试的主要用户路径
- 功能契约摘要
- 浏览器测试证据
- 截图情况
- 可访问性检查
- 视觉层级检查
- 长时间工作体验检查
- 空/加载/错误/成功/禁用/破坏性状态检查
- 自动化检查运行结果
- P0/P1/P2/P3 问题列表
- 已修复问题
- 剩余风险
- 未验证项目
- 后续建议

最终状态只能是以下四种之一：

1. 完成：所有必需检查通过。
2. 带限制完成：仍有非阻塞问题，但已列出。
3. 未完成：仍有 P0 或 P1 问题。
4. 阻塞：必要环境或工具不可用，无法完成验证。

硬规则：

- 如果存在 P0 或 P1，不能写“完成”。
- 如果浏览器测试没有跑，不能写“浏览器已验证”。
- 如果没有截图，不能写“截图已检查”。
- 如果没有 a11y 自动化，不能写“自动化可访问性通过”。
- 如果某项没检查，必须写入“未验证”。
- 不得省略未验证项。

## 辅助脚本

可按需运行 `scripts/run-frontend-qa.sh`。该脚本只读取已有 `package.json` scripts，并尝试运行已经存在的常见检查脚本；它不会安装依赖、修改 lockfile、修改业务文件或生成新的视觉基线。

脚本不能替代真实浏览器 QA、截图审查、可访问性语义审查和人工 UX 判断。

# 智能体类产品试用评估标准（建议稿）

版本：v0.1  
日期：2026-06-11  
适用范围：面向企业内部真实岗位场景的智能体类产品试用评估

## 一、评估原则

本次评估以真实岗位任务完成质量为核心，不以单轮问答效果或模型参数作为唯一依据。参评产品应在统一试用材料、统一任务要求和统一评分口径下接受评估，重点考察产品能否把制度、表单、历史经验、业务工具和审批边界转化为可复核、可采纳、可持续使用的业务成果。

评估应遵循以下原则：

1. **真实场景优先**：以业务人员日常高频任务为主要试用对象，避免只进行开放式闲聊或厂商演示。
2. **结果可复核**：每项任务应保留输入、输出、引用依据、人工修改点、耗时和采纳情况。
3. **业务与技术并重**：技术能力服务于业务闭环，评分重点放在任务完成质量、可信性和试用采纳度。
4. **公平一致**：所有参评产品使用同一批脱敏材料、同一任务说明、同一评分表和同一试用周期。
5. **安全合规**：评估过程不得要求上传涉密、敏感或未经授权的数据；涉及内部资料时应按甲方管理要求脱敏和授权。

## 二、评分总览

满分 100 分。建议业务实际效果相关指标占 65 分，技术与交付支撑指标占 35 分。

| 一级指标 | 分值 | 评估重点 |
| --- | ---: | --- |
| 业务场景理解与适配能力 | 25 | 是否理解真实岗位流程、角色责任、表单字段、风险点和合规边界 |
| 任务闭环与产物质量 | 25 | 是否能完成多步骤任务，并形成可直接使用或少量修改后可使用的成果 |
| 可信性、溯源与幻觉控制 | 15 | 是否引用有据、边界清楚、不编造制度、流程、数据和结论 |
| 知识与工具接入能力 | 10 | 是否支持导入资料、调用工具、沉淀可复用场景能力 |
| 易用性与业务人员采纳度 | 10 | 业务人员是否易上手，输出是否符合甲方语言习惯，是否减少修改成本 |
| 部署适配、安全合规与运维 | 10 | 是否满足终端环境、数据安全、权限、日志、诊断和运维要求 |
| 性能、稳定性与成本 | 5 | 响应速度、长任务稳定性、失败恢复和使用成本是否可接受 |

## 三、详细评分标准

### 1. 业务场景理解与适配能力（25 分）

| 二级指标 | 分值 | 评分说明 |
| --- | ---: | --- |
| 业务意图识别 | 5 | 能否准确识别用户要完成的业务目标，而不是只按字面回答问题 |
| 流程节点理解 | 5 | 能否理解任务所处流程阶段、前后置条件、流转节点和审批边界 |
| 角色责任识别 | 4 | 能否区分发起人、审核人、执行人、监护人、管理人员等不同职责 |
| 表单与材料理解 | 4 | 能否正确理解样例表单、制度条款、历史案例和附件材料之间的关系 |
| 风险与合规意识 | 5 | 能否识别安全风险、合规风险、缺项漏项和不确定内容 |
| 澄清与边界控制 | 2 | 信息不足时是否主动澄清，避免直接编造 |

建议试用场景可覆盖制度问答、流程表单审核、计划编制、缺陷处理、材料输出等典型业务。若涉及工作票、操作票、检修计划等场景，应由甲方提供脱敏样例材料，并以甲方现行制度为准。

### 2. 任务闭环与产物质量（25 分）

| 二级指标 | 分值 | 评分说明 |
| --- | ---: | --- |
| 多步骤任务拆解 | 5 | 能否把复杂任务拆成可执行步骤，并说明当前进度和下一步 |
| 过程可追踪 | 4 | 是否能保留关键过程、判断依据、阶段状态和用户确认点 |
| 输出完整性 | 6 | 产物是否覆盖任务要求，是否遗漏关键字段、风险、责任或结论 |
| 输出可编辑性 | 4 | 产物格式是否清晰，是否便于复制、修改、归档或转成正式文档 |
| 业务可用性 | 4 | 产物是否可直接用于业务办理、审核、沟通、汇报或整改 |
| 异常与返工处理 | 2 | 任务失败、信息缺失或用户调整要求时，是否能恢复和继续 |

评分时应重点查看最终产物是否能减少业务人员实际工作量，而不是只看回答是否流畅。

### 3. 可信性、溯源与幻觉控制（15 分）

| 二级指标 | 分值 | 评分说明 |
| --- | ---: | --- |
| 引用与依据标注 | 4 | 引用制度、规程、历史材料时是否说明来源或条款位置 |
| 事实一致性 | 4 | 输出内容是否与甲方提供材料一致，是否存在张冠李戴或混用口径 |
| 不确定性表达 | 3 | 对缺失信息、过期标准、无法确认事项是否标注待核验 |
| 幻觉控制 | 3 | 是否避免编造政策、流程、设备状态、数据、审批结论或内部规则 |
| 敏感信息处理 | 1 | 是否避免在输出中暴露不应公开的内部信息或敏感字段 |

原则上，不能说明依据的关键结论不得评为高分；对安全、合规、审批类结论，产品应明确区分“已依据材料判断”和“需人工确认”。

### 4. 知识与工具接入能力（10 分）

| 二级指标 | 分值 | 评分说明 |
| --- | ---: | --- |
| 文件与资料接入 | 3 | 是否支持制度文件、表单模板、历史案例、附件材料等资料导入和理解 |
| 场景模板复用 | 3 | 是否能沉淀可复用的任务模板、提示规则、流程步骤或检查清单 |
| 工具与系统扩展 | 2 | 是否具备与本地工具、业务系统、知识库或流程系统对接的扩展能力 |
| 权限与边界控制 | 2 | 工具调用、资料访问和结果输出是否具备权限控制和人工确认机制 |

本项评估采用能力结果口径，不限定具体技术路线。参评产品可通过知识库、场景模板、工具编排、系统接口或其他方式实现。

### 5. 易用性与业务人员采纳度（10 分）

| 二级指标 | 分值 | 评分说明 |
| --- | ---: | --- |
| 上手难度 | 2 | 非技术人员是否能在简短引导后独立完成任务 |
| 交互清晰度 | 2 | 产品是否能清楚说明需要用户补充什么、已完成什么、下一步是什么 |
| 语言与文风适配 | 2 | 输出是否符合甲方业务语境、公文表达和部门沟通习惯 |
| 修改成本 | 2 | 业务人员二次修改量是否明显减少 |
| 采纳反馈 | 2 | 试用人员是否愿意在真实工作中继续使用相关能力 |

建议记录试用人员主观评分、人工修改次数、修改耗时、最终采纳情况和典型反馈。

### 6. 部署适配、安全合规与运维（10 分）

| 二级指标 | 分值 | 评分说明 |
| --- | ---: | --- |
| 终端环境适配 | 2 | 是否符合甲方办公终端、浏览器、操作系统或国产化环境要求 |
| 数据安全 | 2 | 是否能说明数据存储、传输、调用外部模型和敏感资料处理方式 |
| 权限与审计 | 2 | 是否具备用户权限、操作记录、日志留存和必要的审计能力 |
| 故障诊断 | 2 | 出现异常时是否便于定位问题、导出诊断信息和恢复使用 |
| 运维可持续性 | 2 | 是否具备安装、升级、配置、备份、停用和支持服务方案 |

该项应根据甲方信息安全、终端管理和运维制度执行，不限定云端、本地或混合部署形态，但最终方案必须满足甲方数据安全与终端运行要求。

### 7. 性能、稳定性与成本（5 分）

| 二级指标 | 分值 | 评分说明 |
| --- | ---: | --- |
| 响应效率 | 1.5 | 常规任务响应时间是否可接受 |
| 长任务稳定性 | 1.5 | 多步骤、长文本、附件类任务是否能稳定完成 |
| 失败恢复 | 1 | 网络、模型、文件或系统异常后是否能提示原因并继续处理 |
| 使用成本 | 1 | 模型、算力、部署、运维和培训成本是否合理 |

## 四、试用组织方式

### 1. 试用周期

建议试用周期为 7 至 14 天。试用期间业务人员应按真实工作方式使用产品，厂商可提供必要的安装、配置和使用引导，但不应代替甲方完成评分任务。

### 2. 试用角色

| 角色 | 建议权重 | 职责 |
| --- | ---: | --- |
| 业务部门评分人员 | 70% | 评价任务完成质量、业务可用性、修改成本和采纳意愿 |
| 信息化/安全评分人员 | 20% | 评价系统接入、部署适配、数据安全、权限和审计 |
| 运维管理评分人员 | 10% | 评价安装配置、稳定性、故障诊断、升级维护和支持能力 |

### 3. 统一试用材料

甲方宜统一准备以下材料：

1. 脱敏制度文件或管理办法。
2. 脱敏样例表单或流程材料。
3. 脱敏历史案例、问题记录或整改材料。
4. 每项任务的输入说明、目标产物和评分要点。
5. 统一评分表和试用记录表。

### 4. 试用任务范围

每个参评产品至少完成 5 类任务，建议从下列任务中选择：

| 任务类型 | 目标产物 | 重点观察 |
| --- | --- | --- |
| 制度问答与溯源 | 带依据的问答结果 | 是否引用准确、边界清楚、不编造 |
| 流程表单审核 | 审核意见、缺项清单、风险提示 | 是否识别字段、角色、流程和安全风险 |
| 操作步骤校核 | 步骤校核意见、危险点提示、确认清单 | 是否理解顺序、前置条件和人工确认边界 |
| 计划编制 | 工作计划、人员分工、材料清单、风险措施 | 是否能形成可执行计划 |
| 缺陷或问题闭环 | 分级建议、处置流程、整改清单、复盘材料 | 是否能从描述推进到闭环处置 |
| 综合材料输出 | 汇报、纪要、报告初稿或整改说明 | 是否符合甲方语言习惯和正式材料要求 |

### 5. 证据留存

每项任务应至少留存以下证据：

1. 原始输入材料和任务说明。
2. 产品原始输出。
3. 产品引用的依据或来源。
4. 任务耗时。
5. 业务人员人工修改点。
6. 最终是否采纳。
7. 试用人员评分和简要评价。

## 六、结论形成建议

最终结论不宜只给总分排名，应同时给出：

1. 总分和各一级指标得分。
2. 适合优先落地的业务场景。
3. 不适合或需谨慎使用的业务边界。
4. 数据安全、部署运维和系统接入前置条件。
5. 后续试点、扩围或采购建议。

对于得分接近的产品，建议优先比较真实任务采纳率、人工修改成本、可信性问题数量和安全合规风险，而不是单纯比较回答速度或界面观感。

## 七、附：中立措辞要求

为保证评估标准公开、公平、可解释，建议避免使用指向特定厂商或特定实现方式的表述。

| 不建议表述 | 建议表述 |
| --- | --- |
| 必须具备专家团 | 具备多步骤任务协同和过程可追踪能力 |
| 必须支持某某 Skill | 具备可复用场景模板或任务能力沉淀机制 |
| 必须支持某某 MCP | 具备可扩展工具或业务系统接入能力 |
| 必须本地桌面安装 | 符合甲方终端环境与数据安全要求的部署方式 |
| 必须支持工作票、操作票 | 能适配甲方典型流程表单和作业类业务场景 |

## 八、状态边界

本标准为试用评估建议稿。涉及具体制度、流程、表单、系统接口和安全要求时，应以甲方现行制度、脱敏样例材料和信息安全要求为准。

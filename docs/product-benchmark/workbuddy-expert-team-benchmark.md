# WorkBuddy 专家团基准实测记录

记录时间：2026-06-12

## 实测范围

- 应用：WorkBuddy v5.0.3。
- 样例任务：帮我确定新产品的品牌视觉方向，做一个情绪板。
- 输入样例：科技/数码、Z世代/学生、活力·年轻·大胆，后续方向选择为 A 赛博涂鸦。
- 记录方式：通过本机可访问性树和截图观察普通用户界面；未读取 WorkBuddy 源码、配置、令牌或私有数据。

## 状态流转

1. 任务召唤后，WorkBuddy 先显示预计消耗，并进入“深度思考”。
2. 在正式创建团队前，界面显示问答卡，要求用户确认 3 个问题：
   - 新产品类型。
   - 目标受众。
   - 品牌整体感觉。
3. 用户确认后，问答卡变成“已回复”摘要，不再作为普通聊天长文本展开。
4. 系统创建团队 `ai-content-creator-brand-moodboard`，界面显示“已创建”状态。
5. Phase 1 进入“创意策划中”，任务为 `creative-strategist: 品牌视觉策划与情绪板`。
6. 成员条显示创意总监、创意策划师、图文创作专家；当前执行成员有运行态，已完成成员有完成态。
7. Phase 1 完成后，界面展示视觉方向方案，并再次进入用户确认门，询问基于哪个方向制作情绪板。
8. 用户选择 A 赛博涂鸦后，Phase 2 进入图片生成任务，执行成员切换为图文创作专家。
9. 任务完成后，主消息区显示交付总结，产物面板显示 5 张图片和制品列表，团队成员均为已完成。
10. 右侧详情面板提供“产物”和“团队信息”，可查看产物列表与成员状态。

## 对 Taiji 的实现要求

- 专家团状态必须由结构化运行时驱动，不能把系统规则或工具提示词作为用户消息显示。
- 问答确认、团队创建、任务流转、阶段确认、成员状态、产物列表都要有独立 UI 状态。
- 普通用户界面不得显示内部路径、工具名、命令、profile、模型调试信息或旧内部品牌标记。
- 任务重启恢复时必须从持久化 run 状态恢复，而不是从 assistant 文本推断。

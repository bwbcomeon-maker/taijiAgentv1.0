# 我方内部备战清单：智能体类产品试用评估

版本：v0.1  
日期：2026-06-11  
用途：内部准备材料，不作为甲方公开评分标准发布

## 一、备战目标

甲方试用评分标准应保持中立，但我们内部准备要围绕四个优势点展开：

1. **懂业务场景**：能理解工作票、操作票、检修计划、缺陷处理、制度问答等真实流程，而不是只做通用聊天。
2. **能闭环任务**：能把用户一句话需求拆成阶段、产物、风险、下一步和可追踪记录。
3. **产物能用**：输出尽量接近业务人员可直接使用的审核意见、计划、清单、纪要、报告或整改材料。
4. **可落地交付**：能说明终端运行、数据边界、诊断运维、日志脱敏和安全合规方案。

公开标准只写“评价什么”，内部材料要准备“我们如何证明”。

## 二、评分项与我方证据映射

| 评分项 | 对外中立表述 | 我方准备证据 | 试用时要主动呈现 |
| --- | --- | --- | --- |
| 业务场景理解 | 典型业务流程适配能力 | 工作票/操作票/检修/缺陷/制度问答场景包 | 能识别流程节点、角色责任、字段缺项和风险点 |
| 任务闭环 | 多步骤任务协同能力 | 阶段状态、过程记录、产物清单、下一步建议 | 让用户看到不是一次性回答，而是可继续推进的工作台 |
| 产物质量 | 可编辑、可复核、可采纳的业务成果 | 审核意见、整改清单、计划表、纪要、报告初稿样例 | 用“少改就能用”作为业务价值证明 |
| 可信溯源 | 引用依据和不确定性标注 | 制度条款引用、待核验区、风险边界说明 | 避免绝对化结论，主动提示人工确认边界 |
| 知识工具接入 | 可扩展工具/系统接入能力 | 文件导入、附件解析、模板化任务、外部工具接入说明 | 不讲内部技术名词，讲“资料、模板、工具、系统” |
| 易用采纳 | 业务人员低门槛使用 | 试用引导页、示例任务、常用提问模板 | 让用户少学概念，直接按工作任务使用 |
| 部署安全 | 符合终端与数据安全要求 | 本地/内网/混合部署说明、脱敏诊断、权限边界 | 强调数据不乱传、日志可查、问题可定位 |

## 三、五类核心试用场景包

### 1. 制度问答与溯源

**输入材料**

- 一份脱敏制度或管理办法。
- 3 至 5 个业务人员真实会问的问题。
- 1 个信息不足或制度未明确的问题，用来测试边界控制。

**目标产物**

- 带依据的答复。
- 条款出处或材料位置。
- 待人工确认事项。

**我方重点**

- 不只回答结论，要标注依据。
- 对制度没有明确写的内容，输出“需结合本单位细则确认”，不要编造。
- 展示能从长文档中提取相关条款。

**常见失分点**

- 只给泛泛建议。
- 编造制度条款。
- 把不同材料中的口径混在一起。

### 2. 工作票/流程表单审核

**输入材料**

- 一份脱敏样例票据或流程表单。
- 一份相关制度或填写规范。
- 设计 3 至 5 个缺项或风险点，例如字段缺失、职责不清、措施不完整。

**目标产物**

- 缺项清单。
- 风险提示。
- 修改建议。
- 需人工确认项。

**我方重点**

- 识别字段完整性、角色责任、流程前后置条件。
- 将问题分成“必须修正、建议完善、需人工确认”。
- 输出适合业务人员使用的审核意见，而不是技术分析。

**常见失分点**

- 只改文字，不识别业务风险。
- 直接给审批通过/不通过结论，越过人工责任边界。
- 没有说明依据。

### 3. 操作步骤校核

**输入材料**

- 一组脱敏操作步骤。
- 一份操作规程或安全注意事项。
- 一个包含顺序问题或前置条件缺失的样例。

**目标产物**

- 操作顺序校核意见。
- 危险点提示。
- 前置条件和确认清单。
- 人工复核建议。

**我方重点**

- 关注顺序、条件、确认动作和安全边界。
- 对高风险结论使用审慎表达。
- 输出“检查清单”形态，方便用户实际使用。

**常见失分点**

- 把安全类任务当普通文本润色。
- 对关键步骤做确定性判断但无依据。
- 没有保留人工确认环节。

### 4. 检修计划生成

**输入材料**

- 检修对象、时间窗口、人员角色、历史缺陷或风险描述。
- 一份计划模板或历史计划样例。

**目标产物**

- 检修计划初稿。
- 人员分工。
- 材料与工具清单。
- 风险措施。
- 沟通与复盘事项。

**我方重点**

- 把散乱输入整理成结构化计划。
- 体现流程完整性：准备、执行、验收、复盘。
- 输出可复制到正式材料中的段落和表格。

**常见失分点**

- 计划泛泛而谈，没有结合输入材料。
- 忽略人员、时间、材料、风险任一关键维度。
- 输出不可编辑、不可落地。

### 5. 缺陷/问题闭环

**输入材料**

- 一条缺陷描述或问题记录。
- 历史处置样例。
- 分级或闭环管理要求。

**目标产物**

- 问题分级建议。
- 处置流程。
- 责任分工。
- 整改清单。
- 复盘材料初稿。

**我方重点**

- 从“描述问题”推进到“闭环处置”。
- 明确哪些结论是建议，哪些需要责任人确认。
- 形成可追踪清单，便于后续复盘。

**常见失分点**

- 停留在原因分析，没有后续动作。
- 忽略责任人、时限、验证方式。
- 输出没有闭环状态。

## 四、试用引导话术

对甲方业务人员，不讲技术实现，按工作任务引导：

1. “可以把制度、表单或历史材料发给产品，让它先按材料回答，不要先让它自由发挥。”
2. “每个任务都要求产品说明依据、风险点和需要人工确认的地方。”
3. “评分时重点看输出是否减少你的修改量，而不是回答听起来是否聪明。”
4. “如果产品回答不确定，要看它是否主动标注待核验，而不是硬给结论。”
5. “建议保留原始输出和人工修改后的版本，便于比较实际节省时间。”

## 五、公开稿措辞边界

| 内部想表达的优势 | 公开稿应使用的中立表达 | 不建议写法 |
| --- | --- | --- |
| 我们有专家团/阶段化工作流 | 多步骤任务协同和过程可追踪能力 | 必须具备专家团 |
| 我们能做工作票、操作票 | 适配甲方典型流程表单和作业类业务场景 | 必须支持工作票、操作票 |
| 我们能沉淀 Skill | 可复用场景模板或任务能力沉淀机制 | 必须支持 Skill |
| 我们支持工具接入 | 可扩展工具和业务系统接入能力 | 必须支持某某协议或框架 |
| 我们有本地桌面交付 | 符合甲方终端环境与数据安全要求的部署方式 | 必须本地桌面安装 |

## 八、内部复盘口径

试用结束后，内部复盘不要只看总分，要重点看：

1. 哪些任务被业务人员真实采纳。
2. 哪些输出人工修改最少。
3. 哪些场景出现幻觉、引用错误或边界表达不清。
4. 哪些任务用户不知道怎么问，需要改进引导。
5. 哪些部署、安全或运维问题影响继续试点。
6. 哪些能力应该沉淀为固定场景模板。

复盘结论应转化为下一轮产品改进项和演示证据，而不是只写成销售话术。

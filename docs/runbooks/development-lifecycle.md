# 太极 Agent 开发生命周期规范

本文档是本项目唯一的完整开发生命周期规范，覆盖从任务进入、隔离开发、预览与验证，到 PR、CI、合并、清理、回滚和发布交接的全过程。凡涉及仓库文件修改、Git/GitHub、分支/worktree、源码启动、开发与运行目录、持久服务、打包或发布，都必须先阅读并遵循本文。

本文是开发生命周期的 canonical 规范。以下 supporting runbook 只提供受本文约束的动态执行细节，不得覆盖、放宽或改变本文的授权边界、状态口径和门禁；若有冲突，以本文为准：

- PR 分类、`CI Gate` 路由和 GitHub 操作细节：[`github-pr-ci-workflow.md`](./github-pr-ci-workflow.md)
- Kylin/UOS 制包、离线安装与目标机证据：[`taiji-kylin-uos-offline-delivery.md`](./taiji-kylin-uos-offline-delivery.md)

## 1. 核心口径

### 1.1 交付状态不得混用

| 状态 | 最低证据 | 可以宣称 | 不可以宣称 |
| --- | --- | --- | --- |
| 分支已实现 | 功能分支有稳定提交；对应本地验证有记录 | 分支已实现、可供验收 | 已进入正式 `main`、已发布 |
| 已合并 `main` | 正式根目录的本地 `main` 已包含成果；来源与非破坏性复验通过 | 已合并 `main` | 已安装、已发布 |
| 制品已生成 | 已复验 `main` 生成当前候选制品；版本、摘要和构建门禁闭合 | 当前制品已生成 | 已安装、已发布 |
| 安装态已验证 | 该候选制品已在明确环境完成适用的安装、启动、升级/卸载验收 | 该制品在指定环境的安装态已验证 | 已发布、其它环境已通过 |
| 已发布 | 已完成本次定义的外部分发或部署；发布审批、版本/制品身份、适用环境证据和回滚入口闭合 | 已发布到明确渠道、环境和版本 | 未验证环境也已通过 |

“测试通过”“PR 已合并”或 worktree 截图都不能单独跨越上述状态。任何未实时核对的项目必须标记为“未验证”，历史记录只能标记为“历史线索”。

### 1.2 唯一正式代码真相源

- 本地正式根目录的 `main` 是日常启动、打包、发布和正式完成声明的唯一代码真相源。
- 功能分支和 worktree 只用于隔离开发与预览；成果进入 `main` 并从正式入口复验前，只能称“分支已实现”。
- 正常开发不得直接在 `main` 写功能提交，也不得直接向远端 `main` 推送功能提交。
- 默认不 push、不创建 PR、不合并、不发布；外部状态变更必须符合第 4 节的用户授权语义。

## 2. 任务进入与隔离边界

### 2.1 先判断是否会修改文件

| 任务类型 | 分支/worktree 要求 |
| --- | --- |
| 只读排查、解释、审查、状态核验 | 不创建分支或 worktree；在明确仓库路径下只读检查 |
| 需要修改任何仓库文件 | 修改前从最新 `main` 创建短生命周期 `codex/*` 分支和独立 worktree |
| 同一目标的连续多轮调整 | 复用原分支和 worktree，保持同一成果的上下文与历史 |
| 新的独立成果 | 新建一个分支、一个 worktree，最终对应一个 PR |

分支按“独立成果”划分，不按消息数量划分。小改动仍须隔离；它只影响测试分级，不取消分支/worktree 门禁。

### 2.2 修改前的来源核对

修改前必须记录并核对：

```bash
pwd -P
git rev-parse --show-toplevel
git rev-parse --git-common-dir
git branch --show-current
git rev-parse HEAD
git status --short
```

- 基线应来自同步后的 `main`；创建方式和 GitHub 后续命令遵循 [`github-pr-ci-workflow.md`](./github-pr-ci-workflow.md)。
- 已有未提交或未跟踪内容时，先判断归属。当前任务内容应继续收口；无关内容不得覆盖、夹带或擅自清理，必须先提交、隔离或交还其所有者。
- 若物理目录、仓库顶层、Git common dir、分支、HEAD、实际模块来源或运行目录不一致，立即停止实现，先收口边界；在错误来源上得到的验证结果作废。

## 3. 一项成果的状态流转

状态允许暂停和恢复，不要求一次会话走完整条链路。

| 阶段 | 进入条件 | 允许动作 | 离开时应留下的证据 |
| --- | --- | --- | --- |
| 只读核验 | 已确认实际仓库、分支、worktree 与 HEAD | 只读排查、审查和状态核验；不得修改仓库或外部状态 | 实际分支、worktree、HEAD 与核验证据 |
| 本地开发 | worktree 与来源核对完成 | 修改、测试、development 预览 | diff、测试记录、当前状态卡 |
| 等待验收 | 已有可预览结果 | 保持 worktree 与运行边界，等待反馈 | 预览入口、已知限制、下一触发条件 |
| 继续调整 | 收到同一目标的反馈 | 复用原 worktree 迭代 | 更新后的 diff 与验证 |
| checkpoint | 需要安全暂停或保存稳定节点 | 创建本地 checkpoint commit | commit hash、未完成项；不得称完成 |
| Draft PR | 用户已授权提交 PR，但成果仍需协作或验收 | push 分支、创建草稿 PR | PR 地址、未完成项、CI 状态 |
| Ready PR | 验收完成且 PR 可正式审查 | 标记 Ready，等待门禁 | 本地验证、风险分级、审查状态 |
| CI | PR 已触发 `CI Gate` | 修复红灯并重新推送；禁止合并红灯 | 独立干净环境的 CI 结果 |
| 合并 | CI 通过且用户已明确授权合并 | 按专项流程合并 | merge 结果、远端状态 |
| 清理 | 正式 `main` 已同步和复验，任务/进程已结束 | 审计后移除临时 worktree/分支 | 清理审计与干净正式根目录 |
| 发布 | `main` 候选满足对应发布门禁 | 制包、部署、目标环境验收 | 版本/制品/环境绑定证据 |

同一成果可在“本地开发 → 等待验收 → 继续调整”之间多次往返；不因此创建新分支。只读核验不产生交付成果；Draft PR、Ready PR、CI、合并与发布是外部协作阶段，不能由本地完成状态自动推导。

## 4. 用户授权语义

| 用户意图 | 已授权 | 必须明确的对象与边界 | 未授权 |
| --- | --- | --- | --- |
| “本地预览”“给我看看效果” | 在对应 worktree 以 development 模式启动和验证 | 分支、worktree、runtime/config 快照、端口 | push、PR、合并、制包、安装、持久服务、发布 |
| “提交 PR” | 明确暂存、提交、push 功能分支并创建 PR | 分支、目标基线、提交范围、Draft/Ready 状态 | 合并、制包、安装、持久服务、发布 |
| “CI 通过后合并” | PR 通过规定 CI 和全部前置门禁后执行合并，并同步正式 `main` | PR、目标分支、合并方式、回滚入口 | 制包、安装、持久服务、发布 |
| “为 `<commit/版本>` 制包” | 从已复验 `main` 为指定目标生成候选制品 | 源 commit、版本、目标平台、制包环境、输出位置、旧制品处理和回滚边界 | 目标机安装、持久服务变更、分发或发布 |
| “在 `<目标机>` 安装/验收 `<制品>`” | 在指定目标机对指定制品执行约定安装与验收 | 制品版本与摘要、主机/系统、安装/升级/卸载范围、配置与数据影响、备份和回滚边界 | 其它主机、未约定的数据清理、持久服务扩展操作、发布 |
| “在 `<主机>` 安装/启动/停止/卸载 `<持久服务>`” | 对指定服务执行指定生命周期动作 | 服务与 label、版本、主机、安装路径、启停/卸载动作、日志与数据影响、回滚/卸载路径 | 其它服务或主机、全局权限放宽、删除用户配置/会话/日志、发布 |
| “发布 `<版本>` 到 `<渠道/环境>`” | 将指定版本发布到指定渠道或环境 | commit/制品摘要、版本、渠道、环境与受众、覆盖/迁移等破坏性范围、停止和回滚条件 | 其它版本、渠道、环境或扩大投放范围 |
| 未明确说明外部动作 | 在实际工作目录进行只读核验，或在已授权 worktree 本地修改、验证和提交；已有 PR 时可停在 PR | 当前仓库、分支/worktree 与本地影响范围 | push、PR、合并、制包、目标机变更、持久服务变更、发布等外部状态变化 |

- “提交”“保存一下”默认指本地 commit，不自动等于 push 或 PR；语义有歧义且会改变外部状态时必须先确认。
- 用户只授权 PR 时，默认停在 PR；用户只授权合并时，仍必须满足本地验证和 `CI Gate`，不能跳过门禁。
- 创建 PR 时，只有适用本地验证、审查准备和用户验收均已有通过证据，才标记 Ready；状态未知或仍有未完成项时默认创建 Draft。
- 任何授权都只覆盖当前明确成果，不延伸到无关分支、远端 ref、历史改写、部署或发布。
- 制包、目标机安装/验收、持久服务和发布在执行前都必须闭合五类信息：对象、commit/版本或制品摘要、主机/环境/渠道、破坏性范围、备份与回滚边界。任一项缺失时先询问并暂停，不得用历史授权或合理猜测补齐。
- 未获得对应明确授权时，不得创建或改变远端、目标机、持久服务、安装态、发布渠道等外部状态。

## 5. 并行、长期与依赖任务

### 5.1 所有权与互不干扰

- 每个独立成果拥有自己的分支、worktree、进程、依赖目录和状态卡；不得直接修改另一个活动任务的 worktree。
- 并行任务仍在运行时，不得修改、合并、删除、重启或清理其分支、worktree、进程和依赖目录。
- 只有任务明确结束、最终提交稳定、工作树干净、验证证据齐全且相关测试/构建/预览进程退出后，才可进入整合或清理。
- 长期任务使用 checkpoint、Draft PR 和状态卡保存可恢复边界；不得仅依赖聊天上下文或未提交工作树。

### 5.2 依赖与冲突

- 任务 B 依赖任务 A 时，应优先完成并合并 A，再从更新后的 `main` 开始 B。必须并行时，要记录基线、依赖 commit、预期合并顺序和复验范围。
- 两项任务需要修改同一热点文件或同一运行端口时，先明确所有权、启动顺序和集成顺序；无法隔离时暂停后发任务，不在共享目录互相覆盖。
- 合并冲突必须在成果所属分支中理解双方意图后最小化解决，并重跑受影响验证。禁止以覆盖整个文件、丢弃对方变更或改写共享历史代替冲突分析。
- 后续任务基于未合并分支时，前置分支的变更不能被重复包装成独立成果；PR 必须明确依赖关系，合并后重新基于最新 `main` 核对差异。

## 6. 实现规范

### 6.1 通用代码规则

- 修改前先理解现有架构、调用链、测试和运行方式，优先最小改动；禁止夹带无关重构、格式化、依赖升级或产品行为变化。
- Bug 先建立可复现证据，说明根因和影响面；能自动化时先补会失败的回归测试，再修根因并证明测试转绿。不得只压制当前报错或只修表象。
- 新功能先写清验收标准、入口、兼容边界和失败行为；至少覆盖主流程、关键负向场景、既有行为回归，以及该能力实际所在的人工/目标环境门禁。
- 用户可感知能力必须有可见、可发现、可访问的 UI 入口；涉及前端时还须执行项目规定的 `$frontend-ux-qa` 门禁。
- 禁止提交密钥、令牌、私钥、客户数据、用户会话、真实凭据、个人配置、运行日志或临时制品；日志和测试输出不得泄露秘密。
- 不为通过测试而隐式调用系统默认浏览器、真实 OAuth、真实 Provider 或用户日常资料；除非用户明确授权真实验证，否则使用 mock、fixture 和隔离配置。

### 6.2 暂存与提交

- 每次提交前检查 `git diff`、`git status --short` 和将要暂存的文件，只用明确路径暂存；常规开发禁止用宽泛暂存夹带无关内容。
- 提交信息使用 Conventional Commits，例如 `fix: ...`、`feat: ...`、`docs: ...`、`test: ...`、`refactor: ...`；scope 仅在能提升辨识度时使用。
- 一个提交应表达一个可审查目的；checkpoint 也必须说明当前稳定边界，不得伪装为最终完成。
- 代码任务完成并验证通过后必须及时创建本地 commit。若多代理任务已明确由协调者统一提交，执行者不得擅自提交，但必须交付精确 diff、验证结果和脏路径清单。
- 最终报告必须给出本轮 commit hash 与 `git status --short`。无法保持干净时，逐路径说明归属和原因，不得笼统写“有改动”。

## 7. 开发预览与运行来源门禁

### 7.1 development 预览

- 合并前的本地预览必须从该成果对应的 worktree，以项目的 development 模式和该 worktree 的源码入口启动；不得用正式根目录、安装态 App、全局命令或旧构建冒充分支效果。
- 启动和测试必须显式固定源码根、Agent/WebUI/Electron 入口、Python/Node 解释器、runtime-home、用户配置快照和端口，禁止隐式回退到另一 worktree、正式目录、全局安装或旧构建产物。
- 前后端跨目录加载依赖时，显式设置并核对 `HERMES_WEBUI_AGENT_DIR`、解释器路径、`PYTHONPATH` 等来源。在 A 目录执行却加载 B 目录代码时，该轮结果作废，修正环境后重跑。

### 7.2 来源证据

- Python 验证至少记录关键模块的 `__file__`；Node/Electron 至少记录入口真实路径。
- 运行态还要记录进程命令、物理 cwd、监听端口、源码 commit、工作树脏状态、静态资源哈希和配置指纹。
- 自动化测试和常规调试必须 mock 外部打开动作并使用隔离浏览器资料目录。只有用户明确要求真实登录/授权时，才允许打开真实 OAuth 或 Provider 页面。
- 真实桌面验收采用双轨：破坏性场景使用隔离 runtime；用户日常效果使用与正式入口、功能可见性配置等价的快照。隔离 worktree 截图只能证明分支效果，不能代替正式运行目录证据。
- 合并后必须从正式根目录的真实 `main` 启动入口做一次非破坏性复验；安装态和发布态验收另行执行，不能由 development 预览推导。
- 最终交付必须同时报告：`main` commit、正式根目录状态、实际启动入口、进程/模块真实来源、runtime/config 边界及验证结果。任一来源未核对时，对应项目标记为“未验证”。

## 8. 验证与门禁分工

| 门禁 | 目的 | 不能替代 |
| --- | --- | --- |
| 本地测试 | 快速证明改动与受影响回归；提交前必须执行 | CI、真实桌面、目标机 |
| GitHub `CI Gate` | 在独立干净环境按风险复验 | 本地调试、真实 OAuth/Provider、人工视觉、目标机 |
| 人工/真实 Electron | 验证真实入口、交互、窗口与配置边界 | 安装态、Kylin/UOS 目标机 |
| 安装态/发布态 | 验证真实制品、安装路径、升级/卸载和交付证据 | 未测试的平台或客户环境 |

- 文档改动走快车道；普通代码运行根合同和受影响模块；工作流、依赖、认证、凭据、Provider、授权、迁移、打包、发布相关改动自动升级为全量自动化。具体分类以 [`github-pr-ci-workflow.md`](./github-pr-ci-workflow.md) 为准。
- 不确定风险等级时添加 `full-ci` 标签；验证范围只能升级，不能降级。小修改仍走隔离分支，只是可按规则进入快车道。
- 对当前改动适用的门禁是累积证据，彼此不能替代：本地测试不能替代 CI，CI 不能替代本地测试或人工/目标环境验收；人工、真实 Electron、安装态或目标机通过也不能反向替代本地测试与 CI。真实 OAuth/Provider、WPS/Word、Kylin/UOS 目标机验收不进入普通 PR 自动化。
- PR 红灯不得合并；修复后在原分支重新执行本地验证并推送触发 CI。
- 只有日志明确指向 runner 启动/丢失、GitHub 服务事件、网络/依赖源不可用，或测试尚未执行即中断等基础设施证据时，才允许由当前任务负责人对同一 commit 重跑一次，并记录原运行、重跑和判断依据。断言失败、编译错误或可稳定复现的超时不能归类为偶发故障；拿不准时保持失败状态。相同问题重复失败后必须停止重跑，登记并排查 CI 基础设施；修复后仍须取得成功的 `CI Gate`，失败不能当绿灯，也不能合并。
- 未执行的浏览器、截图、可访问性、视觉回归、真实 Provider 或目标环境检查必须标记为“未验证”，不得写成通过。

## 9. PR、合并与正式 `main` 复验

### 9.1 PR 前

1. 核对 diff 只包含当前独立成果。
2. 完成对应本地测试与人工预览，记录未覆盖边界。
3. 明确暂存并创建稳定本地 commit。
4. 按风险分级决定 Draft/Ready 状态和 `full-ci` 标签。
5. 只有用户明确要求“提交 PR”时才 push 并创建 PR；不得直接 push 远端 `main`。

### 9.2 合并条件

- 正常流程中，PR 的唯一合并门禁是 GitHub Actions `CI Gate`；具体合并方式遵循 [`github-pr-ci-workflow.md`](./github-pr-ci-workflow.md)。用户授权、本地验证和审查/验收是进入合并动作的前置条件，不能替代该门禁；下述紧急绕过不等于该门禁通过。
- 必须同时满足：本地验证通过、CI 通过、要求的代码审查已批准且验收已通过、用户已授权合并、依赖 PR 顺序正确。
- 紧急修复仍走 PR。GitHub 不可用时默认停止 push、PR、合并和发布；只有用户本人基于明确截止时间或业务影响判定为紧急，并授权本次要绕过的具体动作与对象，才进入例外评估。例外默认只允许在备份后的本地正式 `main` 整合指定成果，禁止直接向远端 `main` 推送；制包、目标机操作、持久服务或发布仍须另行满足第 4 节对应授权。
- 执行紧急绕过前，带日期的备份标签或 bundle、适用本地验证、回滚命令和完整操作记录必须闭合。GitHub 恢复后，优先把包含紧急整合 commit 的功能/审计分支推送并补建 PR，运行 CI，并采用保留该 commit 为祖先的 merge commit 方式合并；只有远端 `main` 已成为本地紧急 `main` 的后代，才能执行 `git pull --ff-only` 并从正式入口复验。
- 若仓库只允许 squash/rebase，或远端历史变化使紧急 commit 无法保留为祖先，必须停止自动同步：先为本地紧急 `main` 建立 `refs/backup/` 引用和 bundle，再从 `origin/main` 创建专用 reconciliation 分支/worktree，在其中重放或核对紧急成果并重新走 PR/CI。补丁相似不等于满足 fast-forward；只有远端成果已通过下述 exact path/tree/blob/mode proof，或完成逐路径人工等价审查与受影响测试，且正式入口复验通过、备份闭合、用户明确授权重对齐本地 `main` 后，才可执行经审计的非快进重对齐。不得自动 `reset`，补审计失败时停止后续发布并按回滚边界处理。绕过期间不得宣称 CI 已通过，也不能把绕过变成日常流程。

### 9.3 合并后

1. 正式根目录切回 `main`，执行 `git pull --ff-only`。
2. 记录正式 `main` commit 和 `git status --short`。
3. 用祖先关系证明 `main` 包含分支提交；squash/rebase 场景必须使用 exact path/tree/blob/mode proof，或逐路径人工等价审查并重跑受影响测试，证明 `main` 包含该成果。`git patch-id --stable` 最多用于辅助筛查，不能授权完成声明、删除或清理。
4. 从正式入口执行非破坏性复验，重新核对进程/模块来源、runtime/config 边界和必要回归。
5. 只有上述证据闭合，状态才能从“分支已实现”升级为“已合并 `main`”；发布仍需第 12 节证据。

## 10. 清理、导入与回滚

### 10.1 分支/worktree 清理

- 删除前必须审计本地/远端 refs、全部 worktree 状态、未跟踪文件、未提交 diff、仅分支提交、reflog/悬空提交，以及运行/发布目录。
- 任何价值未确认、尚未吸收或仍脏的内容，先建立可回滚备份引用、patch 或归档并记录校验和；不得直接删除。
- 回滚引用统一放在 `refs/backup/`，不能伪装成普通开发分支；至少保留到下一次完整发布验证后 30 天。到期删除前仍须复查归档校验和、悬空提交和远端状态。
- 未完成上述审计时禁止运行 `git gc --prune`。
- 清理必须识别实际合并方式：分支提交已成为 `main` 祖先时可安全删除；squash merge 后不能因 `git branch -d` 失败而机械改用强制删除。自动 exact path/tree/blob/mode proof 必须同时满足：branch base 来自创建分支时记录的基线或已审计的 PR 差异基线，覆盖该成果的完整起点，且是 branch tip 的祖先；squash commit 是正式 `main` 的祖先；以 `--no-renames` 展开的两侧精确变更路径和状态一致；每个涉及路径在 branch tip 与 squash commit 的最终树项存在性、对象类型、文件模式/执行位及 blob/子模块对象 ID 全部一致，删除路径在两侧均不存在。任一不一致、基线来源无法证明、发生 rebase/基线漂移、冲突解决或无法自动比较时，必须逐路径人工等价审查并重跑受影响测试，不得删除。只有上述证明或人工审查、正式入口复验、完整清理审计以及 `refs/backup/` 或等价备份均闭合后，才允许删除原分支；`git patch-id --stable` 不能替代这些门禁。
- 已合并、已复验、干净且无关联进程的临时 worktree/分支应及时清理；正式根目录保留在干净 `main`，不得长期停留在功能分支。

### 10.2 显式仓库路径与 monorepo 导入

- 任何接受显式仓库路径的审计/验收工具，都必须清除 `GIT_DIR`、`GIT_WORK_TREE`、`GIT_COMMON_DIR`、`GIT_INDEX_FILE`、`GIT_OBJECT_DIRECTORY`、`GIT_ALTERNATE_OBJECT_DIRECTORIES` 等环境定位变量，并以“污染环境变量后显式路径仍获胜”的对抗性测试证明不会串库。
- 将独立仓库源码导入 monorepo 时，`git add -A`、工作树干净或文件仍在本机都不是完整性证明；父级或嵌套 `.gitignore` 可能静默吞掉原仓库已跟踪文件。
- 删除旧 Git 元数据前，必须对每个源仓库运行 `scripts/check-imported-source-tree.py`，逐路径证明源 commit 文件已被父仓库跟踪；刚完成导入时必须加 `--require-content-match` 对账 blob 和可执行位。
- 被宽泛 ignore 命中的有效上游文件只能按审计后的精确路径 `git add -f`，不得整体放宽 ignore 规则引入生成垃圾。
- 导入提交后必须从该提交的 `git archive` 或全新 clone 执行一次关键构建/测试，禁止复用 ignored 文件、旧构建目录或嵌套仓库对象制造假绿。

### 10.3 回滚

- 合并后发现问题，优先以 `git revert` 创建可审计回滚，不改写共享历史。
- 回滚也必须走分支、PR、对应测试和 `CI Gate`；紧急程度只能升级响应速度，不能取消证据链。
- 回滚完成后从正式 `main` 入口复验，并在状态卡中说明受影响版本、数据/迁移兼容性和后续修复条件。

## 11. macOS 持久服务边界

macOS `LaunchAgent`、后台守护进程和登录自启动项属于持久运行/部署边界，不属于 development 预览。

- 可执行文件、工作目录、日志和 runtime-home 不得放在 `~/Documents`、开发仓库或任意 worktree；必须使用产品安装目录或用户数据目录，并使用产品专属 label。
- 新增或修改持久服务前，核对实际可执行路径、物理 cwd、日志目录、配置来源、监听端口和卸载路径。
- 安装后必须通过 `launchctl print`、统一日志和进程来源证明服务未被 TCC、旧路径或旧构建劫持。
- 出现 `EX_CONFIG`、反复拉起或无业务日志时，先检查 `xpcproxy`、TCC/沙箱拒绝和 `posix_spawn`；不能只凭退出码修改应用配置，也不能用“授予整个终端/应用完全磁盘访问”掩盖错误部署路径。
- 废弃服务先完成依赖审计和可回滚备份，再 `bootout` 并移出 `~/Library/LaunchAgents`；连续观察确认 label 不再出现、日志不再增长、端口无占用后，才可删除旧运行目录。
- 用户正式配置、会话和日志不是代码垃圾，禁止在服务或 worktree 清理时顺手删除。

## 12. 发布交接

- 发布只能从正式根目录的已复验 `main` 生成；worktree、Draft PR、未合并 commit 或旧制品不得作为发布来源。
- “制品已生成”“安装态已验证”和“已发布”是三个独立证据口径；前一状态不能自动推出后一状态。发布声明必须写明渠道、环境和版本。
- 发布交接至少记录：`main` commit、正式根目录状态、实际启动/制包入口、进程或模块真实来源、runtime/config 边界、本地与 CI 结果、人工/目标环境结果、制品/版本标识、已知风险和回滚入口。
- Linux/Kylin/UOS 打包、安装脚本、桌面壳、诊断、runtime 调整、离线交付和去 Hermes 化必须额外遵循 [`taiji-kylin-uos-offline-delivery.md`](./taiji-kylin-uos-offline-delivery.md) 及项目指定 skill；没有当前制品绑定的目标机证据，不得标记“已发布并验证”。
- 正式交付报告必须明确哪些项目“已实时验证”、哪些“未验证”、哪些仅为“历史线索”。任一来源未核对时，对应结论保持“未验证”。

## 13. 每轮任务状态卡

每次只读核验完成、暂停、交接、请求验收、提交 PR、合并或发布时，都应更新以下状态卡：

```markdown
### 任务状态卡

- 任务：<一个独立成果>
- 分支：<实际被开发或审查的分支；detached HEAD 时记录 HEAD>
- worktree：<实际被开发或审查的 worktree 绝对路径；正式根目录也必须如实记录>
- 阶段：<只读核验 / 本地开发 / 等待验收 / 继续调整 / checkpoint / Draft PR / Ready PR / CI / 合并 / 清理 / 发布>
- 交付口径：<不适用 / 分支已实现 / 已合并 main / 制品已生成 / 安装态已验证 / 已发布>
- 已完成：<已实现且有证据的事项>
- 未完成：<待实现、待验收或未验证事项>
- 验证：<命令、结果、人工/目标环境边界>
- 是否影响 main：<否 / 已合并；附 main commit 与证明>
- 下一触发条件：<用户反馈 / 提交 PR / CI 通过后合并 / 目标机验收等>
```

状态卡必须与事实一致：只读核验的交付口径写“不适用”；只有存在稳定 commit 且适用本地验证通过，才能写“分支已实现”。未提交改动只能处于“本地开发”或“继续调整”；正式 `main` 已包含且复验后写“已合并 `main`”，只有发布证据闭合后写“已发布”。

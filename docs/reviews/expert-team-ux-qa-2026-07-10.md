> 报告基线：2026-07-10 专家团第一性原理审计后的当前工作区实现与本轮验收证据
>
> QA 范围：需求确认浮层、专家团右侧工作台、生成启动与失败恢复、工作台标签页、桌面响应式布局、Electron 源码隔离启动、最终 DOCX/PDF 交付链
>
> 证据原则：本报告严格区分“本轮真实浏览器/真实产物验证”“本轮此前已执行的自动化”“未验证”，不以代码存在、测试字符串、截图或 LibreOffice 转换代替 WPS/Word 终验。

## 状态

**代码合入 GO；完整文档交付验收仍为条件性 NO-GO。**

- 当前已确认的功能缺陷：**P0 = 0，P1 = 0**。在本轮覆盖的主路径中，未复现旧审计中的隐藏二次启动、假“正在生成”、轮询清空输入或失败无恢复入口。
- 当前存在 **1 项 P1 级验收缺口（不是已确认代码缺陷）**：最终 DOCX 尚未在 WPS/Word 真实桌面环境中打开并完成人工目视验收。原因是执行验收时 Mac 处于锁定状态；最终文档交付 gate 因此未关闭。
- 最终相关 fresh gate 已完成：`475 passed, 1 deselected`。被排除用例是仓库既有 Service Worker 品牌名断言，单独执行仍期待 `hermes-shell`，而当前源码已是 `taiji-shell`；该基线问题不冒充本轮通过。
- 结论仅适用于当前 macOS 桌面 WebUI、源码 Electron 隔离环境和本轮四类测试文档，不外推到 Kylin/UOS/Windows、不同 DPI、屏幕阅读器或生产数据规模。

## 变更范围与主要用户目标

本报告审查的是当前实现结果，不在此报告中改动前端业务代码。受影响的主要页面、组件和状态包括：

- 专家团需求确认浮层：3 个必填问题、1 个可选问题、逐题输入、提交与关闭。
- 专家团右侧工作台：待办、协作、成果三个标签页，以及生成中、启动失败和恢复操作。
- 前端状态保持：轮询重渲染时的输入值、焦点、文本选择区、当前标签页和浮层开合状态。
- 后端 mutation 与状态：`/answer`、`/resume`、`generating`、`start_failed`、`version`、`attempt`、`stream` 和 pending 问题数。
- 桌面布局：1120×720、1280×800、1440×920、1920×1080。
- 文档交付：内容创作、研究、会议纪要和含图文档的 DOCX 生成，以及 LibreOffice 转 PDF 后的结构检查。

用户的主要目标是：一次完成必要确认，系统立即、真实地开始当前阶段；失败时能从可见入口恢复；轮询和响应式布局不破坏正在填写的内容；最终拿到可打开、可继续编辑、结构完整的正式文档。

## 主内容 / 辅助内容 / 高级内容

- **主内容**：当前待确认问题、唯一主操作、真实生成状态、失败后的“重新尝试”、当前阶段成果与最终文档入口。
- **辅助内容**：阶段进度、当前成员、工作台标签、生成说明、文档质量提示。
- **高级内容**：`version`、`attempt`、`stream_id`、产物目录、质量报告和运行时身份等内部元数据。此类信息应作为状态合同或诊断证据存在，不应直接以英文内部状态暴露给普通用户。

信息架构判断：需求确认和恢复动作放在右侧工作台或其浮层中，本轮已测的需求确认/启动路径可见；聊天区保留任务上下文，工作台承载状态与操作，成果区承载产物。当前验证未发现已测路径必须依赖隐藏按钮或开发者工具才能完成。

## 真实浏览器测试证据

### 需求确认、轮询与原子启动

本轮在真实桌面 WebUI 中完成 3 个必填问题和 1 个可选问题的逐题填写。浮层保持打开 6.5 秒、经历轮询重渲染后：

- 当前浮层仍然存在并保持打开；
- 已输入的值未丢失；
- 焦点仍位于原输入控件；
- 文本选择起止位置未丢失；
- 没有跳回其它问题或工作台标签。

最后一题提交后，证据来自同一个 `POST /answer` 响应，而不是后续猜测或乐观文案：

```text
state = generating
version = 7
stream = 非空
pending questions = 0
```

随后浮层关闭，右侧工作台显示“正在生成”。这证明本次路径中“完成确认”和“真实创建生成 stream”已成为同一个用户动作；未再出现关闭浮层后寻找第二个启动按钮的旧 P0。

### 错误恢复

将运行置于 `start_failed` 后，工作台显示可发现的“重新尝试”。点击后调用 `/resume`，执行 `attempt` 从 1 增至 2。该路径证明一次启动失败具有可见、可操作的恢复入口；尚未验证连续失败、网络超时、恢复中关闭应用和跨进程重启。

### 键盘操作

- 在标签页上按 `ArrowRight`：移动到下一标签，选中态与焦点同步。
- 按 `End`：移动到最后一个标签，选中态与焦点同步。

本轮没有完成从召唤、全部填写、错误恢复、成果查看到文件打开的全程纯键盘验收，因此不能写成“键盘可访问性全部通过”。

### 响应式 DOM rect 测量

下表是本轮真实浏览器中的 DOM `getBoundingClientRect()` 测量，不是从截图估算的像素，也不是视觉回归基线：

| 视口 | 布局形态 | 专家团工作台高度 | 消息区高度 | 输入区高度 | 遮挡判断 |
|---|---|---:|---:|---:|---|
| 1120×720 | 紧凑上下分区 | 302.4 px | 117.6 px | 186 px | 无 overlap |
| 1280×800 | 紧凑上下分区 | 320 px | 180 px | 未单独记录 | 无 overlap |
| 1440×920 | 左右并排 | 804 px | 602 px | 未单独记录 | 无 overlap |
| 1920×1080 | 左右并排 | 962 px | 760 px | 未单独记录 | 无 overlap |

结论：四档视口均保留可操作的消息区和输入区，工作台没有覆盖主操作。1120×720 下消息区仅 117.6 px，虽未阻塞主路径，但长内容阅读上下文明显偏紧，列为 P2 使用风险而非 P0/P1。

## 发现性、可达性与信息架构

- 正常确认路径的主操作可见；最后确认后浮层自动关闭，不遮挡后续状态。
- `start_failed` 的“重新尝试”是用户可发现入口，不要求通过隐藏快捷键、URL 或开发者工具恢复。
- 工作台待办/协作/成果三个入口可见，并有键盘方向键合同。
- 文件产物的前端合同和真实生成链已存在；本次 Playwright 主路径没有实机点击“打开/下载”最终文件，因此这一步仍单独列入未验证范围。
- 内部 `state/version/attempt/stream` 用于可靠性合同，不应取代用户可理解的中文状态。

在本轮已覆盖能力中，没有发现“代码已有用户能力但页面无可见、可发现、可访问入口”的现存 P1。未覆盖的入口不据此自动判定为通过。

## 可访问性检查

已实时验证：

- 标签页 `ArrowRight`、`End` 切换与焦点同步；
- 轮询后输入焦点和文本选择区保持；
- 浮层不会在当前问题仍待处理时被轮询意外关闭。

本轮前端相关自动化合同 153/153 还覆盖了 `tablist/tab/tabpanel`、`aria-controls`、`aria-labelledby`、`aria-live`、结果查看器 Escape 与焦点返回、Office `locked → ready → consumed` 和 390px 触控目标等代码合同。它证明语义和处理器存在，但不能替代真实辅助技术验收。

明确未验证：

- VoiceOver 或其它屏幕阅读器；
- axe、`@axe-core/playwright`、Lighthouse 自动化可访问性扫描；
- 从召唤到文件打开的全程纯键盘路径；
- 所有弹窗的焦点陷阱、Escape 关闭和关闭后焦点返回；
- 文本/控件对比度、200% 缩放、高对比模式、点击目标尺寸；
- `prefers-reduced-motion`。

结论：**局部键盘与焦点行为通过；自动化可访问性和完整辅助技术验收未验证。**

## 长时间工作体验检查

正向证据：

- 6.5 秒轮询期间没有打断输入、抢焦点或改变选择区；
- 四档桌面视口中工作台没有覆盖消息区或输入区；
- 1440 与 1920 的左右并排布局保留 602 px / 760 px 消息区，适合同时观察对话和工作台。

限制与风险：

- 1120×720 仅保留 117.6 px 消息区，长材料反复对照时可能增加滚动和上下文切换成本；
- 未执行 1 小时以上连续工作、数十次轮询、长列表、长复核项和超长模型输出测试；
- 未检查系统字体替换、125%/150% 缩放、多显示器和低性能设备上的布局稳定性；
- 没有 CPU、内存、重排次数或轮询网络开销基准。

因此不能把短时稳定性写成长时间性能通过。

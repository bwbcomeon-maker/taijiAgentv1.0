## Git hygiene

- 代码改动前先运行 `git status --short`，确认当前工作区状态。
- 如果已有未提交变更，先判断是否属于当前任务；无关变更不得覆盖，必须先收口、提交或隔离。
- 每次代码任务完成并验证通过后，必须及时创建本地 commit；不默认 push，除非用户明确要求或任务明确需要。
- 最终回复必须给出本轮 commit hash 和 `git status --short` 的干净状态；若无法保持干净，必须说明具体剩余路径和原因。

### Canonical main 与 worktree 生命周期

- 本项目唯一正式代码真相源是本地 `main`。功能分支和 worktree 只用于隔离开发，不能作为日常启动、打包、发布或“已经完成”的最终依据。
- 开始修改前必须记录并核对：`pwd -P`、仓库顶层目录、Git common dir、当前分支、HEAD、`git status --short`。发现终端目录、Git worktree、实际模块来源或运行目录不一致时，必须先停止实现并收口边界。
- 在分支完成并验证后，必须先创建本地 commit，再把有效成果安全整合到 `main`，并用祖先关系或等价补丁证明 `main` 已包含该成果。未进入 `main` 的实现只能标记为“分支已实现”，不能标记为正式完成。
- 并行任务仍在运行时，不得修改、合并、删除、重启或清理其分支、worktree、进程和依赖目录。只有任务明确结束、最终提交稳定、工作树干净且相关测试/构建进程退出后，才进入整合。
- 删除分支或 worktree 前，必须完成一次全量审计：本地/远端 refs、所有 worktree 状态、未跟踪文件、未提交 diff、仅分支提交、reflog/悬空提交及运行/发布目录。任何尚未确认价值的内容先建立可回滚备份引用或归档，不得直接删除。
- 已合并且干净的临时分支/worktree应及时清理；正式根目录保留 `main`，不得长期停留在功能分支。默认不 push，除非用户明确要求。

### 开发目录与运行目录一致性门禁

- 测试或启动不能只相信当前 shell 所在目录。必须显式固定源码根、Agent/WebUI/Electron 入口、Python/Node 解释器、runtime-home、用户配置快照和端口，禁止隐式回退到另一 worktree、正式目录、全局安装或旧构建产物。
- Python 验证至少核对关键模块的 `__file__`；Node/Electron 验证至少核对入口真实路径；运行态还要记录进程命令、物理 cwd、监听端口、源码 commit、工作树脏状态、静态资源哈希和配置指纹。
- 前后端联调或测试若跨目录加载依赖，必须显式设置并核对 `HERMES_WEBUI_AGENT_DIR`、解释器路径和 `PYTHONPATH` 等来源。出现“在 A 目录执行、实际加载 B 目录代码”时，该轮结果作废，修正环境后重跑。
- 自动化测试和常规调试默认禁止调用系统默认浏览器、真实 OAuth 登录页或真实 Provider；测试必须显式 mock 外部打开动作并使用隔离浏览器资料目录。只有用户明确要求真实登录/授权时，才允许打开对应外部页面。
- 真实桌面验收采用双轨：破坏性场景使用隔离 runtime；用户日常效果使用与正式入口、功能可见性配置等价的快照，并最终从正式根目录的真实启动入口做一次非破坏性复验。隔离 worktree 的截图不能代替正式运行目录证据。
- 最终交付必须同时报告：`main` commit、正式根目录状态、实际启动入口、进程/模块真实来源、runtime/config 边界及验证结果。任一来源未核对时，相关项目标记为“未验证”。

## 前端 UX QA gate

任何涉及前端、UI、UX、页面、组件、布局、样式、交互、表单、列表、表格、导航、弹窗、可访问性、浏览器测试、截图、视觉优化或功能完整性的任务，都必须显式使用 `$frontend-ux-qa`。

前端任务不能只以“代码能编译”作为完成标准。若代码中存在用户可感知能力但页面没有可见、可发现、可访问的 UI 入口，至少标记为 P1；主流程被阻塞时标记为 P0。没有执行的浏览器测试、截图测试、可访问性自动化或视觉回归必须写为“未验证”，不得写成“通过”。

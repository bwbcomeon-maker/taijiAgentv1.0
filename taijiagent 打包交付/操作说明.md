# 太极 Agent 麒麟终端打包交付操作说明

## 文档分工与四级证据口径

研发和发布负责人应同时阅读源码仓库内的 `docs/runbooks/taiji-kylin-uos-offline-delivery.md`。该内部手册不随精简的现场交付目录单独复制；它沉淀 Docker 演练边界、真实故障、诊断流程、registry lock 风险和目标机最终验收。本文只保留现场操作步骤。

打包和部署状态统一使用以下四级标签：

- **源码包已准备**：当前候选的唯一源码包、basename 校验和源码发布预检通过。
- **制包机已构建**：兼容 Linux amd64 制包机生成 DEB、sidecar、manifest、构建报告、完整离线仓库和 `.build-success`，最终发布预检通过。
- **离线安装已演练**：当前交付目录在断网的干净 Linux amd64 容器、VM 或 chroot 中完成安装、验证、卸载和重装，并生成产物绑定证据。
- **目标机已验证**：真实 Kylin/UOS/openKylin 图形终端完成安装态 Electron、CLI、真实模型、附件、关窗退出和诊断导出验收。

四级证据不能互相替代。看到源码包或 DEB 文件不等于完整离线交付；Docker 通过不等于目标机已验证；旧 commit、旧包、旧日志和截图不能证明当前交付目录已经通过。最终 release 以当前 manifest、两类当轮证据、签名和 `scripts/taiji-release-check.sh` 为准，不以某个历史 commit 名称口头认定。

## 推荐拷贝路径

在本地仓库先执行：

```bash
bash "taijiagent 打包交付/99_本机_准备制包输入包.sh"
```

脚本会刷新当前 commit 对应的唯一源码包、校验 `SHA256SUMS.txt`，并生成：

```text
taijiagent-制包机输入-<hash>.tar.gz
```

这个输入包会排除 macOS 拷贝元数据、旧构建目录和历史安装产物。优先把它拷贝到联网 Linux amd64 制包机，解压后进入 `taijiagent 打包交付` 目录继续执行第一步。

如果不使用输入包，也可以直接把本地的 `taijiagent 打包交付` 文件夹拷贝到联网 Linux amd64 制包机；但拷贝后仍必须先执行制包机脚本，不能直接拿到完全离线目标机安装。

制包成功前，目录里不会有完整 `生成的安装包/` 和 `离线依赖/`；制包成功后，必须把生成后的整个 `taijiagent 打包交付` 文件夹拷贝到离线目标机。

## 完整交付目录

把整个文件夹拷贝到目标终端任意位置：

```text
taijiagent 打包交付
```

不要只拷贝里面的单个脚本。目标机安装阶段的完整文件夹必须包含源码包、制包机脚本、离线依赖仓库、安装验证脚本、诊断脚本和校验文件。

完整交付目录至少应包含：

```text
taiji-agentv1.0-kylin-build-src-<hash>.tar.gz
SHA256SUMS.txt
00_制包机_生成离线交付包.sh
01_制包机_发布预检.sh
02_目标终端_安装并验证.sh
03_目标终端_导出诊断报告.sh
生成的安装包/
离线依赖/
```

只有脚本、说明和源码包不算完整安装包，不能拿到完全离线目标机安装。

## 第一阶段支持边界

- 系统：Kylin V10 SP1，Debian-like
- 架构：x86_64 / amd64
- 包管理器：apt、apt-get、dpkg、systemctl
- glibc：2.31
- 图形桌面：需要能正常启动 Electron 桌面应用

不支持条件：

- RPM-only 终端：需要单独生成 RPM 包，不能用本 DEB 脚本硬装。
- ARM/aarch64 终端：需要单独的 ARM 构建链和 Electron runtime。
- 无包管理器或强隔离终端：后续应走 `.run` fallback，不属于当前 DEB 验收范围。
- 缺少 `离线依赖/Packages` 或 `离线依赖/Packages.gz` 的完全离线交付目录：目标机默认会拒绝安装；新版安装脚本可从 `Packages.gz` 兜底恢复 `Packages`，但正式发布预检要求两个文件都存在。

## 第一步：在联网 Linux amd64 制包机生成离线交付包

在制包机中打开这个文件夹，右键选择“在终端中打开”，然后执行：

```bash
bash ./00_制包机_生成离线交付包.sh
```

这个脚本会自动完成：

- 执行 `01_制包机_发布预检.sh`：检查 git 干净、源码包与当前 commit 匹配、`SHA256SUMS.txt` 只指向当前源码包、交付目录没有旧包/旧 zip/macOS 元数据。
- 如果发现 `.DS_Store`、`._*`、`__MACOSX`、`.AppleDouble` 或 `PaxHeaders*`，脚本会打印路径并自动清理；如果发现旧 zip 或旧安装包，仍会失败，避免误用历史产物。
- 执行管理员能力预检；如果当前用户不能 `sudo`，会在安装构建依赖之前失败并导出失败诊断，不会半跑后才报错。
- 校验制包机架构和包管理器。
- 校验源码压缩包没有传输损坏。
- 解压源码到本文件夹下的 `构建工作区/`。
- 安装构建依赖。
- 安装或复用 `uv`。
- 如果系统 Node/npm 过旧，自动在本文件夹下准备隔离的 Linux x64 Node.js 构建工具。
- 生成 Linux Python venv。
- 通过 npm 获取 Linux Electron runtime。
- 校验默认非密产品配置已进入源码包，且不包含任何敏感凭据。
- 校验 WebUI 运行时资产已本地化，不依赖 jsdelivr/CDN。
- 校验附件上下文能力进入源码包，支持 PDF、PPTX、DOCX、XLSX、TXT/MD/CSV 和图片上传后的对话消费。
- 构建 `amd64 .deb` 安装包。
- 审计最终 DEB payload，确认 Python venv、Electron Linux ELF、Web runtime、脚本和桌面入口都在包内。
- 把生成的安装包复制到本文件夹下的 `生成的安装包/`，并写入 `.build-success` 成功标记。
- 下载 DEB 的直接和递归运行依赖，生成 `离线依赖/Packages` 和 `离线依赖/Packages.gz`。
- 生成 `taiji-package-manifest.json`、构建报告和 SHA256 校验信息。
- 如果任一步失败，会生成 `构建日志/失败诊断-<时间>.txt`，里面包含环境、阶段、关键命令、最近日志和下一步建议。

新版构建会检查 Agent 的产品运行时入口。交付文件夹放在带空格或中文的路径下，也不会因为控制台脚本 shebang 路径解析失败导致双击后 Agent 启动失败。

如果日志里出现下面这种内容，说明旧制包脚本把 Python 依赖同步强制成了 strict，构建会中断，不会有安装文件：

```text
The lockfile at `uv.lock` needs to be updated, but `--locked` was provided.
```

处理方式：重新拷贝当前这份新版交付文件夹，再重新执行第一步；新版脚本默认使用 `TAIJI_UV_LOCK_MODE=auto`，会先按 `uv.lock` 严格同步，遇到制包工作区 lock 漂移时自动重试非 locked 同步并把警告写入日志。如现场只能继续使用旧包，可临时执行 `TAIJI_ALLOW_UV_LOCK_REFRESH=1 bash ./00_制包机_生成离线交付包.sh`，但推荐换用新输入包。

如果日志里出现下面这种内容，说明 WebUI 静态资源还残留运行时 CDN 依赖，完全离线包会被发布门禁拒绝：

```text
WebUI static assets still depend on CDN
```

处理方式：换用当前新版输入包。新版源码已把 PDF.js、Mermaid、xterm、Prism、KaTeX、js-yaml 等运行时静态资产放到 `static/vendor/`，并在本地发布预检中扫描 `cdn.jsdelivr.net`、`unpkg.com`、`cdnjs.cloudflare.com`。

如果制包机能访问 apt 源，但某个 Node/npm/Electron 镜像 DNS 解析失败，脚本会自动切换内置备用源；如果现场网络只能访问指定内网镜像，可在执行前显式指定：

```bash
export TAIJI_NODE_MIRRORS="https://你的内网镜像/node https://nodejs.org/dist"
export TAIJI_NPM_REGISTRIES="https://你的内网npm源 https://registry.npmjs.org"
export TAIJI_ELECTRON_MIRRORS="https://你的内网electron镜像/ https://github.com/electron/electron/releases/download/"
bash ./00_制包机_生成离线交付包.sh
```

制包成功后，你会看到：

```text
生成的安装包/taiji-agent_0.1.0_amd64.deb
生成的安装包/taiji-agent_0.1.0_amd64.deb.sha256
生成的安装包/taiji-package-manifest.json
生成的安装包/构建报告.txt
离线依赖/Packages.gz
离线依赖/Packages
离线依赖/*.deb
```

## 第二步：拷贝到完全离线目标机

把整个 `taijiagent 打包交付` 文件夹拷贝到目标机。目标机不需要联网，不执行 `npm ci`、`uv sync`、`curl` 或在线 `apt update`。

## 第三步：安装并验证

在目标机同一个文件夹中执行：

```bash
bash ./02_目标终端_安装并验证.sh
```

这个脚本会自动完成：

- 检查第一步写入的 `.build-success` 成功标记；没有成功标记不会安装旧包。
- 校验 `taiji-package-manifest.json` 与 `.build-success`、DEB、`离线依赖/Packages.gz` 匹配。
- 校验 `.deb` 文件。
- 检测 `离线依赖/Packages.gz`，在 `/tmp` 下复制出 ASCII 路径的本地 apt 仓库，并同时提供 `Packages` 和 `Packages.gz` 给 Kylin apt 读取，不访问外网。默认缺少该目录会失败；只有明确设置 `ONLINE_OK=1` 才允许使用目标机已有在线源，这不属于完全离线验收路径。
- 如果检测到旧的浏览器 WebUI 版 `taiji-agent`，会自动停止并禁用旧的 `taiji-agent-webui.service` 和 `taiji-agent-gateway.service`。
- 自动清理命令行明确指向 `/opt/taiji-agent` 的旧进程，不按端口盲杀其他软件。
- 自动解除 `taiji-agent` hold 状态，并用 `apt-get purge` / `dpkg --remove` / `dpkg --purge` 清理旧包状态。
- 如果旧包状态始终清不掉，脚本会停止安装新版，避免新旧包状态混在一起。
- 自动删除旧 `/opt/taiji-agent`、旧系统配置、旧 systemd 服务、旧命令入口和旧桌面入口；不需要手动卸载或删除文件。
- 检查 `8787`、`18642`、`18787` 端口，遇到非太极 Agent 进程占用会记录诊断但继续安装；新版桌面端会自动选择空闲本地端口，不会误杀其他软件。
- 使用本地离线 apt 仓库安装太极 Agent。
- 如果机器上已经装过 `taiji-agent`，脚本会自动用当前生成的安装包覆盖安装；遇到同版本重装、apt 判定为“降级”或包被 hold，也不需要手动卸载或删除旧文件。
- 运行安装态诊断命令，并提前验证 Agent CLI 模块入口。
- 验证默认非密配置已同步到当前用户运行态，模型配置和菜单显隐接口可用。
- 在图形桌面会话中尝试 Electron smoke test。
- 检查 `taiji` 命令入口。
- 如果任一步失败，会生成 `构建日志/失败诊断-<时间>.txt`，里面包含目标机环境、包状态、服务状态、端口占用、最近日志和下一步建议。

注意：本脚本按“彻底清除旧版后安装新版”执行，不再备份旧版 `/opt/taiji-agent`。旧 WebUI 版里的模型 Key、微信 token、历史会话和旧运行数据会随旧目录清除；如果还需要旧密钥，只能从旧系统其它记录中另行找回。

## 第四步：双击启动

安装验证脚本通过后，从开始菜单搜索并打开：

```text
太极 Agent
```

验收顺序：

1. 双击“太极 Agent”能打开窗口。
2. 首屏进入对话界面。
3. 再双击一次只聚焦已有窗口，不新开抢端口。
4. 关闭窗口后，本次本地会话停止。
5. 再次双击能重新打开。
6. 首次在界面里配置模型后，发送一句测试消息，确认真实对话返回。
7. 上传一份 PDF/PPTX/DOCX/XLSX 或 TXT/MD/CSV 文件后提问“这个文件说了什么”，回复应基于文件内容，而不是泛泛介绍太极 Agent。
8. 上传一张图片后提问“图里有什么”；如果当前主模型或辅助视觉模型支持图片，应基于图片内容回答；如果未配置视觉能力，应明确提示当前无法分析图片。

Linux 桌面版会隐藏应用菜单栏，不再显示顶部“太极 Agent / 视图”菜单；仍保留麒麟系统原生的最小化、最大化和关闭按钮。

## 第五步：导出诊断报告

如果页面样式、菜单、模型下拉、对话或启动过程仍有异常，在同一个交付文件夹里执行：

```bash
bash ./03_目标终端_导出诊断报告.sh
```

脚本会生成：

```text
诊断报告/taiji-agent-diagnose-<时间>.txt
```

诊断报告会收集包状态、端口、进程、桌面入口、用户态配置摘要、WebUI API 状态、附件解析器状态、图片视觉路由状态和最近日志尾部，并自动脱敏常见密钥字段。后续排查时优先发送这个 txt 文件。

## 重要边界

- 安装包不内置模型 API Key、微信 token、企业微信 Secret、服务器地址或私钥。
- 安装包会内置非密默认产品配置，用来保证菜单显隐、默认模型展示和首页控件与本地产品配置一致；密钥仍需首次进入界面后本机配置。
- 上传附件时，聊天界面只显示文件名，不显示目标机或本机的绝对路径；密钥文件、私钥和疑似密钥文件不会被注入模型上下文。
- 新版安装后只保留开始菜单“太极 Agent”桌面入口；旧浏览器访问版不会作为并存产品保留。
- 安装脚本会删除旧 WebUI 版系统安装和 `/opt/taiji-agent` 运行数据，但不会删除普通用户家目录下的新版用户态目录，例如 `~/.config/taiji-agent`、`~/.local/share/taiji-agent`、`~/.local/state/taiji-agent`。
- 未配置模型前，只能验证桌面壳、本地 Agent/WebUI 和安装链路，不能算真实模型对话已完成。
- 完全离线目标机只安装和验证，不构建、不访问外网。制包机必须使用与目标 Kylin V10 SP1 兼容的 apt 源生成离线依赖。

## 日志位置

如果失败，把下面目录里的日志保留下来：

```text
构建日志/
```

安装后的运行诊断日志在目标机：

```text
~/.local/state/taiji-agent/logs/
```

如果双击后弹出“太极 Agent 启动失败”，新版错误弹窗会显示最近脚本输出；同时保留上面运行诊断日志目录，重点查看桌面端和 Agent 的最近运行日志。

也可以直接执行：

```bash
bash ./03_目标终端_导出诊断报告.sh
```
